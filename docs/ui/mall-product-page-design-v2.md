# C端商城商品页设计文档 v2（评审稿）

## 1. 设计目标与用户路径

### 1.1 设计目标

- 页面定位：C端用户购买入口，不承载运营配置管理功能。
- 核心目标：用户“先浏览商品，再选择拼团/单买，再支付完成”。
- 体验目标：页面结构清晰、视觉简洁、文案精炼、接口失败可降级。
- 技术目标：复用现有后端接口，不引入复杂前端框架，便于快速落地。

### 1.2 用户路径

1. 用户进入商城页（默认登录后落地）。
2. 浏览商品列表，选择目标商品。
3. 查看拼团优惠与可参与团队。
4. 选择“参与拼团/开团购买/单独购买”。
5. 进入支付弹窗并确认支付。
6. 支付后触发结算与回调流程，完成购买。

## 2. 页面信息架构（模块分区）

页面建议固定为 6 个模块，严格按优先级从上到下布局：

1. `Header` 顶部信息区
   - 品牌/商城名
   - 运营看板入口（次要）
2. `ProductCatalog` 商品浏览区
   - 多商品卡片横滑/网格
   - 商品摘要：名称、卖点、价格
3. `ProductDetail` 商品详情区
   - 主图
   - 标题与简短副标题
   - 优惠信息与价格对比
4. `TeamList` 拼团列表区
   - 在拼团队伍卡片
   - 剩余人数、倒计时、参团按钮
5. `ActionBar` 底部固定购买区
   - 单独购买按钮
   - 开团购买按钮
6. `PaymentModal` 支付弹窗
   - 支付金额
   - 交易单号
   - 二维码
   - 取消/支付完成

说明：
- 旧版冗余模块（过长介绍、大量说明文字、非核心营销话术）不进入首屏。
- 运营看板作为“辅助入口”，不影响商城主流程。

## 3. 视觉风格规范（参考运营看板）

### 3.1 色彩规范

- 页面背景：`#f4f7fb`
- 卡片背景：`#ffffff`
- 主文字：`#1f2937`
- 次文字：`#6b7280`
- 主按钮色：`#2563eb`
- 危险/价格强调：`#dc2626`
- 成功状态：`#16a34a`
- 边框：`#e5e7eb`

### 3.2 字体与层级

- 主标题：20px / 700
- 模块标题：16px / 600
- 正文：14px
- 次要信息：12px

### 3.3 间距与圆角

- 页面主间距：12px~16px
- 卡片内边距：12px
- 卡片圆角：12px
- 小元素圆角：8px

### 3.4 组件样式

- 卡片统一阴影：轻阴影（看板同级）
- 按钮统一风格：
  - 主按钮：实心主色
  - 次按钮：浅色背景 + 主色文字
- 状态色：
  - 倒计时/优惠：主色或强调色
  - 失败提示：危险色

## 4. 关键交互流程

### 4.1 选商品

- 点击商品卡片 -> 更新当前商品状态
- 同步刷新：主图、标题、副标题、价格、拼团列表

### 4.2 参与拼团 / 开团

- 点击团队中的“参与拼团” -> 带 `teamId` 调用锁单
- 点击“开团购买” -> `teamId = null` 调用锁单
- 锁单成功 -> 展示支付弹窗
- 锁单失败 -> Toast/Alert 提示错误码及信息

### 4.3 单独购买

- 点击“单独购买” -> 直接展示支付弹窗
- 支付完成 -> 调用单买回调测试接口

### 4.4 支付完成与结算

- 拼团支付完成 -> 调用结算接口
- 结算成功 -> 页面刷新
- 结算失败 -> 明确错误提示

### 4.5 失败兜底

- 商品配置加载失败：展示默认价格与兜底文案
- 拼团列表加载失败：显示可读空态
- 全局 Promise 异常：统一兜底提示

## 5. 多商品数据模型设计

### 5.1 前端商品模型（建议）

```json
{
  "id": "p1",
  "goodsId": "9890001",
  "activityId": 100123,
  "name": "微服务架构实战课程",
  "subtitle": "从0到1搭建可落地交易系统",
  "image": "./images/sku-13811216-01.png",
  "originalPrice": 129,
  "groupPrice": 59
}
```

### 5.2 运行态状态模型（建议）

```json
{
  "userId": "u10001",
  "selectedProductId": "p1",
  "outTradeNo": "",
  "payMode": "group",
  "prices": {
    "origin": 129,
    "group": 59,
    "deduction": 70
  }
}
```

说明：当前后端样例可能只有单一 `goodsId`，首版可先前端模拟多商品，后续再接真实多 SKU。

## 6. API 对接草案（请求/响应映射）

### 6.1 查询营销配置

- URL: `/api/v1/gbm/index/query_group_buy_market_config`
- Method: `POST`
- Request:

```json
{
  "userId": "u10001",
  "source": "s01",
  "channel": "c01",
  "goodsId": "9890001"
}
```

- Response（示意）：

```json
{
  "code": "0000",
  "data": {
    "goods": {"originalPrice": 129, "payPrice": 59, "deductionPrice": 70},
    "teamList": [],
    "teamStatistic": {"allTeamUserCount": 100}
  }
}
```

### 6.2 锁单

- URL: `/api/v1/gbm/trade/lock_market_pay_order`
- Method: `POST`
- Request 核心字段：`userId/teamId/activityId/goodsId/outTradeNo/notifyUrl`

### 6.3 结算

- URL: `/api/v1/gbm/trade/settlement_market_pay_order`
- Method: `POST`
- Request 核心字段：`source/channel/userId/outTradeNo/outTradeTime`

### 6.4 单买回调（测试）

- URL: `/api/v1/test/direct_buy_notify`
- Method: `POST`

## 7. 响应式规则

- `> 760px`：详情区左右布局（图 + 信息）
- `<= 760px`：详情区上下布局（图在上、信息在下）
- 商品卡片区保持横向滚动，确保多商品浏览效率
- 底部购买栏固定，始终可见

## 8. 验收标准与实施里程碑

### 8.1 验收标准

- 页面结构与视觉符合文档规范（卡片化、简洁、低文字密度）
- 用户可完成完整链路：选商品 -> 开团/参团 -> 支付 -> 结算
- 接口失败时有可读降级，不出现空白按钮/空白模块
- 移动端与桌面端都可正常操作

### 8.2 里程碑

1. M1 - 设计评审确认（本文件）
2. M2 - 静态页面重构（HTML/CSS/JS）
3. M3 - 后端接口联调（锁单/结算/回调）
4. M4 - 体验验收与简历文案同步
